# Choice Problem Audit: E-Commerce Benchmark Datasets

## Executive Summary

A revealed-preference graph is only meaningful if each observation represents a genuine choice occasion. For budget-based analysis, that means one observation must correspond to one budget set and one chosen bundle at credible prices. For menu-based analysis, one observation must correspond to one menu of genuinely available alternatives and one chosen item. If the observation unit is wrong, the preference graph is wrong, and everything downstream (consistency tests, efficiency scores, predictive features) inherits that error.

This report audits all eight benchmark configurations across seven datasets currently used in the PrefGraph e-commerce benchmark suite. For each dataset, we document exactly what the raw data contains, how it is transformed into choice observations, and whether the resulting observations constitute defensible choice occasions. Where data was available locally, we sampled and inspected individual observations to verify whether they make intuitive sense as economic choices.

The central finding is that only three of the eight benchmarks produce observations that unambiguously look like real choice occasions: Dunnhumby (budget), REES46 (menu), and to a lesser extent Taobao (menu). Two more are salvageable with modifications (H&M, Instacart Menu). The remaining three should be dropped or fundamentally reconceived.

## Part 1: What Makes a Valid Choice Occasion

### Budget Case (GARP/CCEI/MPI)

In the budget model, a consumer faces a price vector **p** and chooses a quantity bundle **x** subject to the budget constraint **p . x <= m**. Revealed preference theory tests whether a set of such observations (p_1, x_1), (p_2, x_2), ..., (p_T, x_T) could have been generated by a utility-maximizing consumer.

For this test to be meaningful, each observation must satisfy three requirements:

1. **Coherent shopping episode.** The observation must represent a single decision context. Aggregating purchases across a week or month is standard (Beatty and Crawford 2011 use weekly aggregation for the same Dunnhumby data), but the aggregation period must be short enough that the consumer plausibly faced a stable set of prices and made a deliberate allocation.

2. **Common goods universe.** All observations for a given consumer must share the same set of goods. If observation 1 includes "milk" but observation 2 does not, the revealed-preference relation between them is undefined. In practice, this means fixing a set of commodity categories and treating zero purchases as "chose zero units at the prevailing price."

3. **Credible price vector.** The prices must reflect what the consumer actually faced. If the price vector is imputed (for example, a median across all consumers), the budget constraint p . x is only as meaningful as the imputation. If prices are constant across time, the budget lines are all parallel, budget sets never properly intersect, and GARP violations become impossible regardless of the consumer's actual behavior. This is exactly what happened with Instacart.

### Menu Case (SARP/WARP/Houtman-Maks)

In the menu model, a consumer faces a finite set of alternatives (the menu) and picks one. Revealed preference theory says that if the consumer chose item a when item b was available, then a is revealed preferred to b. SARP requires that these revealed preferences form no cycles.

For this test to be meaningful:

1. **Simultaneous availability.** The menu must represent items that were genuinely available to the consumer at the same time. A sequential feed of recommendations (where the consumer sees items one at a time, in an order determined by an algorithm) does not constitute a menu in the economic sense.

2. **Single chosen item.** The consumer must select exactly one item from the menu. If the consumer selects multiple items (as in a grocery basket), the observation does not fit the standard menu-choice framework without additional assumptions about which item was "preferred."

3. **Exogenous menu construction.** The menu should not be determined by the consumer's own past behavior in a way that makes violations trivially impossible or trivially guaranteed. For example, if the menu is always the same set of items, SARP violations are impossible with a consistent chooser regardless of preferences.

## Part 2: Dataset-by-Dataset Audit

### Dataset 1: Dunnhumby ("The Complete Journey")

#### Raw Data

The Dunnhumby dataset is a publicly available record of grocery purchases from a major US retail chain, covering approximately 2,500 households over a two-year period (104 weeks). The raw data consists of two files:

- **transaction_data.csv**: Each row is a single item purchased. Columns include `household_key` (anonymous household identifier), `DAY` (integer 1 through ~728, representing calendar days), `PRODUCT_ID`, `QUANTITY` (number of units purchased), `SALES_VALUE` (total dollar amount paid), `RETAIL_DISC` (store discount applied), `COUPON_DISC` (manufacturer coupon discount applied), and `STORE_ID`.

- **product.csv**: Maps each `PRODUCT_ID` to a `COMMODITY_DESC` string (e.g., "FLUID MILK PRODUCTS", "SOFT DRINKS", "COLD CEREAL").

The data contains real dollar amounts at the item level. Discounts are separately recorded, allowing computation of the effective unit price paid.

#### Observation Construction

The loader (`src/prefgraph/datasets/_dunnhumby.py`) transforms this transaction-level data into weekly budget observations as follows:

1. **Commodity selection.** Ten commodity categories are hardcoded: Soft Drinks, Fluid Milk, Bread, Cheese, Bag Snacks, Soup, Yogurt, Beef, Frozen Pizza, and Lunchmeat. All transactions not in these categories are discarded.

2. **Unit price computation.** For each transaction, the unit price is computed as `(SALES_VALUE - RETAIL_DISC - COUPON_DISC) / QUANTITY`. Prices outside the $0.01 to $50.00 range are clipped before further aggregation.

3. **Weekly price oracle.** For each commodity and each week, the median unit price across all households and all transactions in that commodity-week cell is computed. This produces a 104-by-10 price matrix shared by every household.

4. **Forward/backward fill.** If a commodity has no transactions in a given week (and therefore no median price), the missing value is filled by carrying forward the most recent observed price, then backward-filling any remaining gaps at the start of the series.

5. **Quantity aggregation.** For each household, week, and commodity, quantities are summed. If a household bought no items in a commodity during a given week, the quantity is zero.

6. **User filtering.** Households with fewer than 10 active weeks (weeks in which they purchased at least one item from the 10 categories) are excluded.

The result is that each observation is one week for one household: a price vector of length 10 (shared across all households in that week) and a quantity vector of length 10 (specific to that household in that week).

#### Assessment

**Prices.** The shared price oracle is the principal caveat. All households face the same prices in a given week, which ignores household-specific coupon usage, store-level pricing differences, and promotional targeting. However, this approach has direct precedent in the empirical revealed preference literature: Dean and Martin (2016) use median market prices for exactly this type of analysis. Large US grocery chains do maintain approximately uniform shelf pricing across stores in a metropolitan area, so the assumption is defensible even if imperfect.

**Quantities.** These are real unit counts aggregated from transaction-level data. A household that bought 3 units of Fluid Milk and 2 units of Cheese in week 12 will have quantities [3, 2] in those positions. This is a direct measurement, not an imputation.

**Goods universe.** Ten commodities is small, covering roughly 15 percent of a typical household's weekly grocery budget. The remaining 85 percent of spending is unobserved. This is a limitation of the analysis scope, not a defect in the choice problem construction per se. The interpretation is that we are testing rationality of allocation across these 10 categories specifically, not across the household's full consumption bundle.

**One-sentence defense:** "Each week, a household allocates grocery spending across 10 commodity categories at the chain-wide median shelf price for each category."

**Verdict: DEFENSIBLE.** This is the strongest budget dataset in the benchmark suite. The observation unit (weekly household allocation) is standard in the literature, the prices are derived from real transaction values, and the quantities are direct measurements. The shared price oracle and the narrow goods universe are acknowledged limitations, not fundamental flaws.

**Data availability note:** Dunnhumby data was not downloaded on the local machine at the time of this audit, so no individual observations were sampled. The assessment is based on code inspection and the dataset's established use in the revealed preference literature.

### Dataset 2: Open E-Commerce (Amazon Purchase History)

#### Raw Data

The Open E-Commerce dataset is a crowdsourced collection of Amazon purchase histories, where users voluntarily shared their order data. The raw file `amazon-purchases.csv` contains one row per item purchased, with columns including `Survey ResponseID` (anonymous user identifier), `Order Date`, `Category` (a freeform text string from Amazon's product taxonomy), `Purchase Price Per Unit` (in US dollars), `Quantity`, and `Title` (product name).

The category field is unstructured: it contains Amazon's hierarchical category strings, which can be highly specific (e.g., "Electronics > Computers & Accessories > Laptop Accessories > Laptop Bags & Cases") or very broad.

#### Observation Construction

The loader (`src/prefgraph/datasets/_open_ecommerce.py`) performs the following transformation:

1. **Category mapping.** The freeform category strings are mapped to a smaller set of standardized categories using keyword matching. For example, any category containing "kindle" or "book" maps to "Books"; any containing "electronic" or "computer" or "phone" maps to "Electronics"; any containing "clothing" or "shoes" maps to "Clothing." Categories that match no keyword are assigned to "Other." After mapping, the top 50 categories by transaction count are retained.

2. **Monthly price oracle.** For each standardized category and each calendar month, the median purchase price across all users is computed. Missing months are forward-filled, backward-filled, and finally filled with the category's global median. If no data exists at all for a category, the fallback is $0.01.

3. **Quantity aggregation.** For each user, month, and category, quantities are summed.

4. **User filtering.** Users with fewer than 5 active months are excluded.

#### Assessment

**Prices.** The shared price oracle is fundamentally problematic for Amazon data. Amazon's pricing is notoriously dynamic and personalized. The same product can have different prices for different users based on geographic region, Prime membership status, seller competition, algorithmic pricing experiments, and coupon/promotion targeting. Computing a single median price per category per month across all users and treating it as the price every user faced is a strong assumption that is unlikely to hold.

Moreover, the category aggregation exacerbates the problem. "Electronics" includes items ranging from $5 USB cables to $2,000 laptops. The median price of "Electronics" in a given month is not a meaningful number that any individual user faced as a per-unit price for the electronics "good."

**Quantities.** These are real purchase counts, which is fine.

**Goods universe.** The keyword-based category mapping is lossy. Kindle books are merged with physical books. The "Other" bucket contains everything that did not match any keyword, which could be a large and heterogeneous set. The top-50 selection is data-driven but the categories themselves are noisy.

**One-sentence defense:** "Each month, a user allocates Amazon spending across 50 product categories at global median prices identical across all users."

**Verdict: WEAK.** The core problem is that Amazon does not have uniform pricing. The shared price oracle assumption that works tolerably for a grocery chain (Dunnhumby) breaks down for an e-commerce marketplace with algorithmic and personalized pricing. A reviewer would reasonably ask: "If User A paid $30 for a book and User B paid $15 for the same book, why are you assigning both of them the median price of $22?" The budget constraint computed with imputed prices does not reflect the actual trade-offs any individual user faced.

**Data availability note:** Not downloaded locally; assessment based on code inspection.

### Dataset 3: H&M (Fashion Recommendations)

#### Raw Data

The H&M dataset comes from a Kaggle competition on personalized fashion recommendations. The raw file `transactions_train.csv` contains one row per article purchased (not per transaction), with columns `t_dat` (date), `customer_id` (hashed string), `article_id` (8-digit integer), `price` (floating-point, normalized to a 0-1 scale), and `sales_channel_id` (1 for online, 2 for in-store).

This is the largest dataset in the benchmark suite by transaction volume: approximately 31.8 million rows covering 1.36 million customers over 24 months (September 2018 to September 2020).

The critical detail is that **each row represents one article unit purchased.** If a customer buys two identical scarves in one visit, there are two rows with the same `article_id`, date, and customer. We verified this empirically: in a sample of 1 million rows, 8.5 percent of (date, customer, article) combinations appear more than once, with a maximum of 100 duplicates. The rows are individual article units, not basket-level aggregates.

#### Observation Construction

The loader (`src/prefgraph/datasets/_hm.py`) transforms this data as follows:

1. **Product group assignment.** The first two digits of the `article_id` are used as a product group identifier. This maps roughly 100,000 distinct articles into approximately 20 product groups.

2. **Top-K selection.** The 20 most frequent product groups by total transaction count are retained; all others are discarded.

3. **Monthly price oracle.** For each product group and each month, the median transaction price across all customers is computed. The 4-level fallback chain is: forward-fill, backward-fill, product-group median, global median, and finally $0.01.

4. **Quantity aggregation.** For each customer, month, and product group, the number of article units purchased is counted (`groupby().size()`). Because each row in the raw data is one article unit, this count represents the number of articles purchased, which is a valid quantity measure.

5. **User filtering.** Customers with fewer than 6 active months are excluded. The maximum number of users is capped at 50,000 (the most active by transaction count).

#### Sampled Observations

We loaded 3 users and inspected the first user's data:

```
User customer_49beaacac0c7: 25 months, 9 active product groups

Month 1 | Normalized expenditure: 0.1144
  Group 3: 3 articles at price 0.0381

Month 2 | Normalized expenditure: 1.2585
  Group 0: 1 article at price 0.0339
  Group 1: 33 articles at price 0.0254
  Group 3: 14 articles at price 0.0254
  Group 4: 3 articles at price 0.0102

Month 3 | Normalized expenditure: 1.8471
  Group 0: 6 articles at price 0.0373
  Group 1: 54 articles at price 0.0254
  Group 3: 11 articles at price 0.0229
```

Month 2 to Month 3 shows this customer increasing their purchase of Group 1 articles from 33 to 54 (a 64% increase) while reducing Group 3 purchases from 14 to 11. The prices for Group 1 remained stable (0.0254) while Group 3 prices decreased slightly (0.0254 to 0.0229). This is the kind of substitution pattern that GARP is designed to test: did the customer's reallocation across product groups respect the budget constraint as relative prices shifted?

#### Assessment

**Prices.** H&M has real transaction prices, which is the best of any budget dataset in the suite. However, they are normalized to a 0-1 scale with no documented conversion back to absolute currency. This means relative price comparisons across product groups are valid (Group 0 is roughly 50% more expensive per article than Group 4), but absolute expenditure figures are meaningless. The shared price oracle (median across all customers) is the same caveat as Dunnhumby. For H&M, this is moderately defensible: H&M maintains relatively uniform pricing across stores and online, with less personalization than Amazon.

**Quantities.** Initially flagged as a potential "transaction count" bug, but verified to be correct. Each row in the raw CSV is one article unit. The `groupby().size()` operation counts article units, not transactions. A customer who bought 54 articles in Group 1 in a month did indeed purchase 54 physical garments (or accessories, or home items) in that product group.

**Goods universe.** Twenty product groups derived from the first two digits of the article ID is a coarse grouping. A "product group" might contain everything from basic t-shirts to designer jackets. The within-group heterogeneity is higher than Dunnhumby's commodity categories.

**One-sentence defense:** "Each month, a customer allocates fashion spending across 20 product groups at real (normalized) transaction prices."

**Verdict: MODERATE.** H&M's choice problem is defensible. The quantities are real article counts, the prices are derived from actual transactions, and the monthly observation unit is reasonable for fashion retail. The main limitations are the normalized price scale (no absolute dollars), the coarse product grouping, and the shared price oracle. None of these are fundamental flaws; they are standard trade-offs in empirical revealed preference work.

### Dataset 4: Instacart (Budget Version)

#### Raw Data

The Instacart Market Basket Analysis dataset contains 3.2 million grocery delivery orders from 206,000 users across approximately 50,000 products in 134 aisles and 21 departments. The raw files include:

- **orders.csv**: `order_id`, `user_id`, `eval_set`, `order_number` (sequential per user), `order_dow` (day of week), `order_hour_of_day`, `days_since_prior_order`
- **order_products__prior.csv**: `order_id`, `product_id`, `add_to_cart_order` (integer indicating the sequence in which the user added items to their cart), `reordered` (0 or 1)
- **products.csv**: `product_id`, `product_name`, `aisle_id`, `department_id`

Critically, **there is no price column anywhere in the Instacart dataset.** The dataset records what users ordered but not what they paid.

#### Observation Construction

The budget-based loader (`src/prefgraph/datasets/_instacart.py`) assigns heuristic prices to each of the 134 aisles using keyword matching on the aisle name. For example, aisles containing "spirit" or "alcohol" are assigned $14.00 per unit; aisles containing "fresh" and "produce" are assigned $2.50; "meat" aisles get $6.00. These prices are constant across all users and all time periods. The price matrix is constructed via `np.tile(price_vector, (T, 1))`, which replicates the same 134-element price vector for every observation.

#### Assessment

This construction is fatally flawed and has already been abandoned. With constant prices across all observations, the budget lines for any two observations are parallel. Parallel budget lines can never properly intersect, which means the mathematical conditions required for a GARP violation can never be triggered. We verified this empirically on 200 users: every single user is perfectly GARP-consistent, with CCEI = 1.0, MPI = 0.0, and zero violations. Fourteen of the 31 revealed preference features have exactly zero variance across all users.

The budget-based Instacart benchmark produces revealed preference features that are guaranteed to be uninformative by construction. It does not test whether RP features predict future behavior; it tests what happens when your feature set is a vector of constants.

**Verdict: DEAD.** No amount of fixing can salvage this. The raw data simply does not contain prices, and any price imputation that is constant across time will produce the same degeneracy.

### Dataset 5: Instacart (Menu Version)

#### Raw Data

Same raw files as above. The menu-choice version additionally uses `departments.csv` to scope the analysis to individual grocery departments.

#### Observation Construction

The menu-choice loader (`src/prefgraph/datasets/_instacart_menu.py`) constructs observations as follows:

1. **Department scoping.** For each user, the department in which they have the most reorders is identified. Only products from this department are considered.

2. **Growing known set.** The loader walks through the user's orders chronologically. For each order t, the "menu" is defined as the set of all products the user has ever purchased in prior orders (1 through t-1) within the selected department.

3. **Choice identification.** Among the products in order t that are in the known set (i.e., reorders, not new products), the one with the minimum `add_to_cart_order` value is designated as the "choice."

4. **Filtering.** Only observations where the menu contains between 3 and 30 items are kept. Users with fewer than 5 valid observations are excluded.

#### Sampled Observations

We inspected User 75124, who has 100 orders and whose most active department is dairy eggs. This user is a yogurt enthusiast. Their orders look like this:

**Order 1** (first order, all items are new): 20 yogurt products spanning multiple brands (Total, Fat Free, Almond Milk, O'Soy). Cart positions 1 through 20 are all yogurt.

**Order 2**: 16 products, of which 11 are reorders and 5 are new. Cart position 1 is Sparkling Water (a beverages item, cross-department). Cart positions 2 through 16 are dairy eggs reorders.

**Order 3**: 27 products, of which 22 are reorders. Cart position 1 is Green Tea (beverages, cross-department). Cart positions 4 through 27 are dairy eggs items.

**Order 4**: 17 products, all reorders from dairy eggs.

After the loader processes this user, the observations look like:
- Observation 1: Menu of 23 previously-purchased yogurt products. Choice: product 7 (whichever yogurt the user added to cart first among the reorders).
- Observation 2: Menu of 24 products. Choice: product 14.
- Observation 3: Menu of 24 products. Choice: product 5.

The fundamental problem is visible in the raw data: **this user purchases 15 to 20 yogurt products in every single order.** The "choice" is simply whichever yogurt happened to be added to the cart first. This user is not choosing one yogurt from a set of 24 alternatives; they are buying almost all of them every time. The `add_to_cart_order` signal likely reflects the order in which items appear in the user's "Buy Again" list in the Instacart UI, or the order in which aisles are encountered during a virtual grocery run, not a genuine preference ranking.

For SARP analysis, this means: if in Order 5 the user adds "Strawberry" first and in Order 8 they add "Blueberry" first (while buying both in both orders), the system records a "preference reversal" that is really just a UI ordering difference.

#### Assessment

**Menu.** The "growing known set" construction is defensible in concept. The idea that a user's effective choice set for a reorder consists of products they have previously tried is reasonable and mirrors how people actually shop for habitual purchases.

**Choice.** The "first item added to cart among reorders" choice signal is the critical weakness. In a standard menu-choice experiment (or even in natural comparison shopping like REES46's sessions), the consumer views multiple alternatives and selects one, leaving the others unchosen. In this Instacart construction, the consumer purchases most or all of the menu items, and we designate one of them as the "choice" based on cart ordering. This is not what menu-choice theory is designed to analyze.

**One-sentence defense:** "When reordering within a grocery department, a user picks one product first from their set of previously purchased alternatives."

But the honest version is: "When reordering 15 to 20 yogurts from a habitual shopping list, the cart order in which they are added is treated as a preference ranking."

**Verdict: WEAK.** The menu construction is sound, but the choice signal is noise. This benchmark can be salvaged if the choice signal is redesigned. Possible alternatives include: restricting to orders where the user reorders exactly one product from the department (analogous to REES46's single-purchase filter), using brand-switching events as choice signals, or using new-product adoption as a choice signal.

### Dataset 6: REES46 (Multi-Category E-Commerce)

#### Raw Data

The REES46 dataset is a publicly available record of user behavior on a large multi-category e-commerce platform, covering October and November 2019. The raw files `2019-Oct.csv` and `2019-Nov.csv` contain approximately 110 million event rows with the following columns:

- `event_time`: Timestamp (UTC) to the second
- `event_type`: One of "view", "cart", or "purchase"
- `product_id`: Integer product identifier
- `category_id`: Integer category identifier
- `category_code`: Hierarchical string (e.g., "electronics.smartphone", "appliances.environment.air_heater")
- `brand`: String brand name (e.g., "samsung", "lg", "pituso")
- `price`: Floating-point price in USD
- `user_id`: Integer user identifier
- `user_session`: UUID string assigned by the platform's server

The critical column is `user_session`. This is not a heuristic reconstruction from timestamps; it is the platform's own session identifier, assigned by the server when the user begins a browsing session and terminated when the session ends according to the platform's own criteria.

#### Observation Construction

The loader (`src/prefgraph/datasets/_rees46.py`) constructs menu-choice observations:

1. **Event filtering.** Only "view" and "purchase" events are retained. Cart events are dropped.

2. **Session grouping.** Events are grouped by the platform-defined `user_session` identifier.

3. **Single-purchase filter.** Only sessions containing exactly one "purchase" event and at least one "view" event are kept. Sessions with zero purchases, multiple purchases, or no views are discarded.

4. **Menu construction.** The menu is the set of all product IDs that had a "view" event in the session, plus the purchased product (which may or may not have been explicitly viewed).

5. **Menu size filtering.** Sessions where the menu contains fewer than 2 or more than 50 items are discarded.

6. **User filtering.** Users with fewer than 5 valid sessions are retained. Item IDs are remapped to a per-user 0-indexed namespace.

#### Sampled Observations

We reconstructed three actual sessions from the first 500,000 rows of the October file:

**Session 1 (User 541146978):**
- Menu: LG TV at $694.67, Samsung smartphone at $383.51
- Choice: Samsung smartphone
- Interpretation: User looked at a TV and a phone, bought the phone. Cross-category browsing ending in a deliberate purchase.

**Session 2 (User 555538567):**
- Menu: Pituso kids' swing at $90.04, Elenberg air heater at $33.44, Tefal air heater at $90.07, Timberk air heater at $64.33, unbranded air heater at $36.53
- Choice: Pituso kids' swing
- Interpretation: User viewed the swing four times (suggesting strong interest), then browsed air heaters (perhaps comparison shopping for a separate need), and purchased the swing. The repeated views of the swing suggest it was the primary target.

**Session 3 (User 531760733):**
- Menu: Vivo smartphone at $231.64, Samsung smartphone at $253.33
- Choice: Samsung smartphone
- Interpretation: Textbook comparison shopping. User viewed two competing smartphones and purchased one. This is exactly the kind of observation that menu-choice theory is designed to analyze.

#### Assessment

**Menu.** Platform-defined sessions are the gold standard for session identification. The server knows when a user's browsing session begins and ends, which eliminates the ambiguity inherent in heuristic session boundary detection. The menu (set of viewed products within a session) is a reasonable proxy for the user's active consideration set during that browsing episode.

**Choice.** The single-purchase filter ensures that each session produces exactly one chosen item. This is clean and unambiguous.

**Caveats.** The main limitation is impression bias: the menu contains only items the user actually viewed, not the full catalog of available products. If a user would have preferred a product they never saw, that preference is unobservable. This is an inherent limitation of observational (as opposed to experimental) menu-choice data and applies to all non-experimental datasets.

Additionally, the menu includes all viewed products regardless of category. Session 1 (TV and smartphone) and Session 2 (swing and heaters) involve cross-category items that a consumer may not have been actively comparing. However, the fact that these items were viewed within the same session suggests at least some consideration, and the single-purchase filter ensures the outcome is unambiguous.

**One-sentence defense:** "In a platform-defined browsing session, a user views several products and purchases exactly one."

**Verdict: STRONG.** This is the strongest menu dataset in the benchmark suite. Platform-defined sessions, real product and price data, single-purchase filter, and sampled observations that look like genuine comparison shopping. The impression bias caveat is acknowledged but standard.

### Dataset 7: Taobao (Alibaba User Behavior)

#### Raw Data

The Taobao dataset contains approximately 100 million user behavior events from the Alibaba/Taobao e-commerce platform. The raw file `UserBehavior.csv` has no header row. The columns, inferred from the loader code, are:

- Column 0: `user_id` (integer)
- Column 1: `item_id` (integer)
- Column 2: `category_id` (integer)
- Column 3: `behavior_type` (string, one of "pv" for page view, "buy" for purchase, "cart" for add-to-cart, "fav" for favorite/wishlist)
- Column 4: `timestamp` (Unix timestamp in seconds)

Unlike REES46, there is no platform-defined session column. Sessions must be inferred from the timestamp data.

#### Observation Construction

The loader (`src/prefgraph/datasets/_taobao.py`) constructs menu-choice observations:

1. **Event filtering.** Only "pv" (page view) and "buy" events are retained. Cart and favorite events are dropped.

2. **Session detection.** Events are sorted by (user_id, timestamp). A new session begins whenever the user_id changes or the gap between consecutive events exceeds 1,800 seconds (30 minutes). Session IDs are assigned as the cumulative sum of these session-break indicators.

   The 30-minute threshold was chosen based on exploratory data analysis (EDA) of inter-event gap distributions. The EDA found that 84 percent of inter-event gaps are shorter than 30 minutes, with a sharp distributional break at the 90th percentile (approximately 3.3 hours). The 30-minute cutoff captures the natural cluster of within-session activity while separating it from between-session gaps. Calendar-day boundaries were explicitly rejected because they split genuinely continuous browsing sessions that span midnight.

3. **Single-purchase filter.** Only sessions with exactly one unique purchased item are kept.

4. **Menu construction.** Same as REES46: the set of all viewed (pv) items in the session, plus the purchased item.

5. **Filtering.** Menu size 2 to 50; minimum 5 sessions per user.

#### Sampled Observations

We reconstructed three sessions from the first 1 million rows:

**Session 1 (User 100, duration 28.4 minutes):**
- Menu (viewed): items 3658601, 5153036, 3245421 (three items across two categories)
- Choice: item 598929 (a fourth item not in the viewed set, from a third category)
- Interpretation: The purchased item was not viewed before purchase. This means the "menu" was artificially extended by the union operation: the loader adds the purchased item to the viewed set to ensure the choice is in the menu. But the user never actually viewed this item as part of their browsing, at least not in the logged data. This could indicate a direct purchase (user knew exactly what they wanted and navigated directly to it) or a data gap (the view event was not logged).

**Session 2 (User 117, duration 62.9 minutes):**
- Menu: 10 items, all in category 1575622
- Choice: item 4552317 (purchased at 03:38, continued browsing until 04:27)
- Interpretation: This is excellent comparison shopping data. The user browsed 10 items in the same category over about an hour, purchased one partway through, and continued browsing (perhaps comparing what they bought against alternatives, or exploring additional options after committing). The session duration exceeds 30 minutes because the gap between consecutive events never exceeds 30 minutes, even though the total session spans 63 minutes. This is correct behavior for the gap-based session rule.

**Session 3 (User 122, duration 5.4 minutes):**
- Menu: 3 items across 3 categories
- Choice: item 646921
- Interpretation: Short, focused session. User viewed three items in different categories and bought one. Plausible as a quick purchase occasion.

#### Assessment

**Session definition.** Gap-based session detection is a well-established methodology in web analytics and e-commerce research. The 30-minute threshold is within the standard range used in the literature (typically 20-30 minutes). The EDA validation adds credibility. However, it remains a heuristic: a 25-minute or 35-minute cutoff would produce different session boundaries, and there is no ground truth to calibrate against.

**Menu.** The viewed-items-as-menu construction has the same impression bias as REES46. Additionally, Session 1 reveals a specific issue: the purchased item is sometimes not in the viewed set, which means the loader's union operation is masking a data quality problem. We cannot determine how common this is without a broader analysis, but it suggests that some "menus" are partially synthetic (the menu does not reflect items the user actually considered).

**Choice.** The single-purchase filter is clean when the purchased item was indeed browsed alongside alternatives (Session 2). It is less meaningful when the purchase was a direct navigation with no comparison shopping (Session 1).

**One-sentence defense:** "In a 30-minute browsing session, a user views several products and purchases exactly one."

**Verdict: MODERATE.** Taobao produces some excellent choice observations (Session 2 is as good as anything in REES46) but also some questionable ones (Session 1, where the purchased item was never viewed). The gap-based session heuristic is standard but not as clean as REES46's platform-defined sessions. Overall defensible, with the caveat that a non-trivial fraction of observations may involve purchases not preceded by browsing.

### Dataset 8: Tenrec (Tencent QQ Browser Video)

#### Raw Data

The Tenrec dataset is a video recommendation dataset from Tencent's QQ Browser, released as part of a NeurIPS 2022 Datasets and Benchmarks submission. Two files are available:

- **QK-video.csv** (14 GB): Full dataset with approximately 5 million users and 140 million interactions.
- **QB-video.csv** (77 MB): Small sample with approximately 34,000 users.

The columns are: `user_id`, `item_id` (video identifier), `click` (binary: 1 if user clicked the video, 0 if it was shown but not clicked), `follow` (binary: 1 if user followed the video's creator), `like` (binary: 1 if user liked the video), `share` (binary: 1 if user shared the video), `video_category`, `watching_times`, `gender`, `age`.

The data records interactions between users and a recommendation algorithm. Each row represents a video that was shown to the user. The `click` column indicates whether the user engaged with it, and the `like`/`follow`/`share` columns indicate deeper engagement.

#### Observation Construction

The loader (`src/prefgraph/datasets/_tenrec.py`) constructs menu-choice observations using a "feedback window" approach:

1. **User iteration.** For each user, the loader walks through their rows in chronological order (the rows are assumed to be temporally ordered in the source file).

2. **Window accumulation.** Each row's item is added to a running "window" of items shown since the last feedback event.

3. **Window closure.** When a row has feedback = 1 (by default, the "like" column), the current window is closed as one observation: the menu is the set of all items in the window, and the choice is the item that received the like.

4. **Menu size filtering.** Windows with fewer than 2 or more than 50 items are discarded.

5. **User filtering.** Users with fewer than 5 valid windows are retained.

#### Sampled Observations

We inspected three users from the QB-video.csv sample:

**User 105030 (501 total clicks, 10 likes = 2% like rate):**
- Window 1: 8 items shown, user liked the 8th. Menu size: 8. Reasonable.
- Window 2: **265 items shown**, user liked the 265th. Menu size: 265 (exceeds the 50-item cap and would be filtered, but the raw window is this large). Between likes, this user scrolled through 265 videos. This is not "choosing from a menu." This is passively consuming a recommendation feed and occasionally reacting.
- Window 3: 1 item shown, user liked it immediately. Menu size: 1. Degenerate (a menu of one item has no choice content).

**User 143895 (464 clicks, 229 likes = 49% like rate):**
- Window 1: 2 items. Liked the 2nd. Barely non-degenerate.
- Window 2: 1 item. Liked it immediately. Degenerate.
- Window 3: 1 item. Liked it immediately. Degenerate.
- Pattern: This user likes almost every other video. The resulting "menus" are overwhelmingly of size 1 or 2. There is essentially no choice content.

**User 200533 (74 clicks, 7 likes = 9.5% like rate):**
- Window 1: 12 items, liked the 12th. Moderate.
- Window 2: 9 items, liked the 9th. Moderate.
- Window 3: 6 items, liked the 6th. Moderate.
- Pattern: This user has a moderate like rate, producing windows of reasonable size.

#### Assessment

The Tenrec dataset fails the fundamental requirement of a menu-choice problem on multiple grounds.

**Not a simultaneous menu.** In a menu-choice setting, the consumer faces a set of alternatives presented simultaneously and selects one. In Tenrec, videos arrive one at a time in a sequential feed controlled by a recommendation algorithm. The user never sees the "menu" as a set; they see a stream of items, and at some point they react positively to one. The items shown at position 1, position 50, and position 200 in the stream were not simultaneously available alternatives that the user evaluated and ranked.

**"Like" is not "choose."** Liking a video does not mean preferring it over all other videos in the window. It means the user had a positive reaction to that specific video. The user may have enjoyed (but not liked) several earlier videos. The absence of a like does not mean "not preferred"; it may mean "not sufficiently motivated to tap the like button."

**Extreme window size variance.** Window sizes range from 1 (degenerate, no choice content) to 265+ (absurd as a "menu"). The min/max filter (2-50) catches the worst cases, but a window of 50 sequential recommendations is still not a meaningful choice set.

**High like rates produce degenerate data.** User 143895 likes 49% of videos, producing mostly size-1 and size-2 windows. These observations cannot contribute to SARP analysis (you need at least 3 items in at least 2 observations to create a SARP cycle).

**Algorithmic menus.** The videos shown to each user are selected by Tencent's recommendation algorithm. SARP violations may reflect changes in the recommendation algorithm (which alters the "menu") rather than inconsistencies in the user's preferences. The benchmark documentation already acknowledges this, stating that the analysis measures "user-algorithm interaction consistency" rather than pure user preferences. But this reframing undermines the benchmark's purpose: if we are not measuring preference consistency, then testing whether "preference consistency features" predict future behavior is circular.

**One-sentence defense:** "Between consecutive likes, a user clicks through several recommended videos and likes exactly one." But this sentence reveals the problem: it describes a process (sequential consumption with intermittent feedback), not a choice (deliberate selection from simultaneously available alternatives).

**Verdict: DEAD.** This is not menu-choice data. It is sequential engagement data from a recommendation feed. The observation construction imposes a menu-choice interpretation on data that does not have that structure. Sampled observations confirm the theoretical concern: 265-item windows, 49% like rates, and size-1 menus are incompatible with the discrete-choice framework that SARP analysis requires.

## Part 3: Consolidated Findings

### Defensibility Summary

| Dataset | Type | Verdict | Critical Issue |
|---------|------|---------|----------------|
| **Dunnhumby** | Budget | DEFENSIBLE | Shared price oracle (standard caveat) |
| **REES46** | Menu | STRONG | Impression bias only (standard caveat) |
| **Taobao** | Menu | MODERATE | Some purchases not in viewed set; heuristic sessions |
| **H&M** | Budget | MODERATE | Normalized prices; coarse product groups; shared oracle |
| **Instacart Menu** | Menu | WEAK | Bulk reorder; "first in cart" is noise, not preference |
| **Open E-Commerce** | Budget | WEAK | Shared price oracle indefensible for Amazon |
| **Tenrec** | Menu | DEAD | Sequential feed, not simultaneous menu; degenerate windows |
| **Instacart Budget** | Budget | DEAD | Constant prices; zero variance in all RP features |

### What Should Change

**Drop immediately (no salvage path):**
- Instacart Budget: No prices in the raw data. Any constant imputation makes GARP trivially satisfied.
- Tenrec: The data structure (sequential recommendation feed) is fundamentally incompatible with menu-choice theory. No change to the loader can fix this because the problem is in the data, not the code.

**Drop or reframe with strong caveats:**
- Open E-Commerce: Amazon's personalized pricing makes the shared price oracle assumption indefensible. If retained, it must be explicitly framed as a sensitivity analysis testing what happens when the price oracle is known to be wrong, not as evidence that RP features work or fail.

**Salvageable with modified choice construction:**
- Instacart Menu: The menu (known products from prior orders) is sound. The problem is purely the choice signal. Restricting to orders where the user reorders exactly one product from the department (analogous to REES46's single-purchase filter) would produce clean menu-choice observations. The trade-off is that the sample size will shrink dramatically.

**Keep as-is (possibly with documentation improvements):**
- Dunnhumby: Standard in the literature. Shared price oracle is an acknowledged limitation.
- H&M: Real per-article prices and valid quantity measures. Normalized price scale and coarse grouping are acceptable trade-offs.
- REES46: Strongest menu dataset. Platform-defined sessions and clean single-purchase filter.
- Taobao: Moderate quality. Gap-based sessions are standard. Some noise from purchases not in viewed set.
